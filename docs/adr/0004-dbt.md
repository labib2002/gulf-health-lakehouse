# ADR 0004 — Transform layer with dbt (Postgres)

- **Status:** Accepted
- **Phase:** 3 — dbt
- **Date:** 2026-04-25

## Context

Raw synthetic tables land in Postgres (`raw` schema). We need a maintainable,
tested transform layer that turns them into an analytics-ready **star schema**
(facts + conformed dimensions), with lineage and data-quality tests, on
`dbt-postgres`. Targets: 10+ models, 20+ tests, deliberate materializations
incl. an incremental model, and `dbt docs` lineage.

## Decision

**Layered project** (`transform/dbt/`):

- **staging/** (7 models, **views**): one model per source, light cleaning only —
  typing, renaming, lowercasing `meal_type`. Views are cheap and always reflect
  the latest raw load.
- **intermediate/** (2 models, **ephemeral**): reusable building blocks
  (`int_nutrition_daily` rolls meals → per-day macros; `int_activity_enriched`
  adds derived metrics). Ephemeral = inlined as CTEs, no warehouse object to
  manage.
- **marts/** (5 models, **tables**): conformed `dim_user`, `dim_device`; facts
  `fct_daily_activity`, `fct_body_scan`, `fct_nutrition`. Tables because they're
  queried often by BI.
- **`dim_date`** as a committed **seed** (CSV date spine) — deterministic and
  shared across facts via `date_key`.

**Materializations are deliberate, not default** (set in `dbt_project.yml`):
view (staging) · ephemeral (intermediate) · table (marts) · **incremental**
(`fct_daily_activity`). The fact grows by date, so it's incremental on
`activity_date` with `unique_key=(user_id, date_key)` and `delete+insert`; a
re-run inserts 0 rows (verified) instead of rebuilding 270k.

**Testing** (59 data tests): generic `unique`/`not_null`/`relationships`/
`accepted_values`, `dbt_utils.accepted_range` and
`unique_combination_of_columns`, a **custom generic test** (`positive`), and two
**singular tests** (`assert_sleep_stages_within_total`,
`assert_experiment_balanced`).

**Profiles** are self-contained and secret-free (env-var driven), targeting the
compose Postgres on host port 5433; a `bigquery` output is stubbed for Phase 5.

## Alternatives considered

- **All marts as views** — rejected; facts are large and re-queried, so tables
  (and one incremental) pay off.
- **`fct_daily_activity` as a full table** — rejected; it's the one model whose
  size and date-append pattern actually justify incremental, which is the point
  to demonstrate.
- **Snapshots for the current user attributes** — deferred to Phase 5 (BigQuery)
  where SCD2 is the explicit deliverable; on Postgres `dim_user` keeps the
  current tier via a window function.
- **`merge` incremental strategy** — `delete+insert` is simpler and supported on
  Postgres; `merge` is a BigQuery-side choice in Phase 5.

## Consequences

- `dbt build` is green: 14 models, 1 seed, 59 tests, PASS=72 total.
- `dbt docs generate` produces the lineage graph (`target/index.html`).
- The same models re-target BigQuery in Phase 5 with minimal change.
- intraday_hr is loaded as a source but not yet modeled here — it's the heavy
  input for the Phase 7 Spark job.

## Interview check

**Staging model vs mart?**
Staging is a thin, 1:1-with-source cleaning layer (typing/renaming, no business
logic, materialized as views) that isolates the rest of the project from raw
quirks. Marts are the business-facing star schema (facts at a grain + conformed
dimensions), materialized as tables for query performance.

**Ephemeral vs view vs table vs incremental — and why each here?**
- *view* (staging): cheap, always fresh, no storage; fine for light transforms.
- *ephemeral* (intermediate): inlined CTEs, no object to manage; good for
  internal building blocks reused by one or two downstream models.
- *table* (marts): materialized once, fast to query; right for BI-facing models.
- *incremental* (`fct_daily_activity`): only process new rows since the last run
  — essential when a fact grows by date and a full rebuild is wasteful. Verified
  the re-run inserts 0 rows; `--full-refresh` rebuilds when logic changes.
